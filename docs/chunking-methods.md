# 多种分块方式功能说明

## 功能概述

系统现已支持 6 种不同的文档分块策略，用户可以根据文档类型和需求选择最合适的分块方式。

## 分块方式详解

### 1. 智能分块 (Intelligent Chunking)

**适用场景**：结构化文档（有标题、章节的文档）

**工作原理**：
- 自动识别文档结构（标题、段落、列表）
- 按章节和段落进行分块
- 保持语义完整性

**识别的结构**：
- Markdown 标题（# ## ###）
- 章节标题（第一章、第二节）
- 编号标题（1. 2. 一、二、）

**优点**：
- 保持文档结构
- 语义完整性好
- 适合技术文档、书籍

**示例**：
```
第一章 系统架构
  1.1 前端架构
  1.2 后端架构
第二章 数据库设计
  2.1 表结构
  2.2 索引设计
```
每个章节会被识别为独立的分块单元。

---

### 2. 问答分块 (Q&A Chunking)

**适用场景**：FAQ 文档、问答对文档

**工作原理**：
- 识别问答对格式
- 每个问答对作为一个分块
- 保持问题和答案的关联

**支持的格式**：
```
Q: 如何安装系统？
A: 执行以下步骤...

问：系统支持哪些功能？
答：系统支持...

1. 如何配置数据库？
   首先需要...
```

**优点**：
- 问答对完整性
- 检索精确度高
- 适合客服知识库

---

### 3. 表格分块 (Table Chunking)

**适用场景**：包含大量表格的文档

**工作原理**：
- 识别 Markdown 表格
- 每个表格作为独立分块
- 保持表格结构

**支持的格式**：
```markdown
| 列1 | 列2 | 列3 |
|-----|-----|-----|
| 值1 | 值2 | 值3 |
```

**优点**：
- 保持表格完整性
- 适合数据密集型文档
- 便于表格数据检索

**注意**：
- 目前仅支持 Markdown 格式表格
- PDF 中的表格会被转换为文本

---

### 4. 通用分块 (General Chunking)

**适用场景**：纯文本文档、无明显结构的文档

**工作原理**：
- 固定大小分块（默认 512 tokens）
- 固定重叠（默认 64 tokens）
- 简单快速

**优点**：
- 处理速度快
- 适用范围广
- 实现简单

**缺点**：
- 可能在句子中间截断
- 语义完整性较差

---

### 5. 父子分块 (Parent-Child Chunking)

**适用场景**：需要高精度检索和完整上下文的场景

**工作原理**：
- 两级分块策略
- 子分块（512 tokens）：用于向量检索
- 父分块（1536 tokens）：提供完整上下文

**数据流程**：
```
文档
  ↓
父分块（1536 tokens）
  ↓
子分块（512 tokens）
  ↓
向量化（仅子分块）
  ↓
检索时：匹配子分块，返回父分块
```

**优点**：
- 检索精度高（小块匹配）
- 上下文完整（大块返回）
- 平衡性能和质量

**缺点**：
- 存储开销大（两级分块）
- 处理时间较长

**推荐使用**：这是系统默认策略，适合大多数场景。

---

### 6. 递归分块 (Recursive Chunking)

**适用场景**：通用文档处理

**工作原理**：
- 按分隔符层级递归切分
- 分隔符优先级：段落 → 行 → 句子 → 词 → 字符
- 保持语义完整性

**分隔符层级**：
1. `\n\n` - 段落分隔
2. `\n` - 行分隔
3. `. ；。` - 句子分隔
4. 空格 - 词分隔
5. 字符级切分（最后手段）

**优点**：
- 避免在句子中间截断
- 支持中英文混合
- 语义完整性好

## 使用流程

### 1. 上传文档

文档上传后，状态为 **"待分块"**（parsed），不会自动分块。

### 2. 选择分块方式

在文档列表中，为每个文档选择合适的分块方式：

| 文档类型 | 推荐分块方式 |
|---------|------------|
| 技术文档、书籍 | 智能分块 |
| FAQ、客服文档 | 问答分块 |
| 数据报表 | 表格分块 |
| 纯文本 | 通用分块 |
| 通用场景 | 父子分块 |
| 长文本 | 递归分块 |

### 3. 执行分块

**单个文档**：
1. 选择分块方式
2. 点击"执行分块"按钮
3. 等待处理完成

**批量执行**：
1. 勾选多个文档
2. 在底部批量操作栏选择分块方式
3. 点击"批量执行分块"
4. 系统按顺序处理每个文档

### 4. 查看结果

分块完成后：
- 状态变为 **"完成"**（done）
- 显示分块数量
- 可点击"查看分块"查看详情

## API 接口

### 单个文档分块

```http
POST /api/v1/kb/{kb_id}/documents/{doc_id}/chunk?chunk_method=intelligent
```

**参数**：
- `chunk_method`: 分块方式（intelligent, qa, table, general, parent_child, recursive）

**响应**：
```json
{
  "id": "doc-uuid",
  "filename": "document.pdf",
  "status": "done",
  "chunk_method": "intelligent",
  "total_chunks": 45
}
```

### 批量分块

```http
POST /api/v1/kb/{kb_id}/documents/chunk-batch?chunk_method=parent_child
Content-Type: application/json

{
  "doc_ids": ["doc-uuid-1", "doc-uuid-2", "doc-uuid-3"]
}
```

**响应**：
```json
{
  "processed": 3,
  "results": [
    {
      "id": "doc-uuid-1",
      "filename": "doc1.pdf",
      "status": "done",
      "chunk_method": "parent_child",
      "total_chunks": 30
    },
    {
      "id": "doc-uuid-2",
      "filename": "doc2.pdf",
      "status": "done",
      "chunk_method": "parent_child",
      "total_chunks": 25
    },
    {
      "id": "doc-uuid-3",
      "filename": "doc3.pdf",
      "status": "failed",
      "error_message": "No content extracted"
    }
  ]
}
```

## 状态说明

| 状态 | 说明 | 可执行操作 |
|------|------|-----------|
| pending | 等待上传 | - |
| parsing | 解析中 | - |
| parsed | 待分块 | 选择分块方式并执行 |
| chunking | 分块中 | - |
| done | 完成 | 查看分块、删除 |
| failed | 失败 | 重新执行分块、删除 |

## 最佳实践

### 1. 根据文档类型选择

```
技术文档 → 智能分块
FAQ 文档 → 问答分块
数据报表 → 表格分块
混合文档 → 父子分块
```

### 2. 批量处理相同类型文档

对于同一类型的多个文档，使用批量执行功能可以提高效率。

### 3. 测试不同策略

对于重要文档，可以尝试不同的分块策略，比较检索效果。

### 4. 监控分块结果

分块完成后，查看分块详情，确认分块效果是否符合预期。

## 常见问题

### Q1: 如何选择合适的分块方式？

**A**: 参考上面的"推荐分块方式"表格。如果不确定，使用"父子分块"（系统默认）。

### Q2: 可以重新分块吗？

**A**: 可以。删除文档后重新上传，或者在未来版本中支持重新分块功能。

### Q3: 批量执行会很慢吗？

**A**: 系统按顺序处理每个文档，处理时间取决于文档数量和大小。建议分批处理大量文档。

### Q4: 分块失败怎么办？

**A**: 查看错误信息，常见原因：
- 文档内容为空
- 文档格式不支持
- 分块参数不合理

### Q5: 不同分块方式可以混用吗？

**A**: 可以。每个文档可以使用不同的分块方式，系统会正确处理。

## 数据库迁移

如果从旧版本升级，需要执行数据库迁移：

```sql
-- 添加 chunk_method 字段
ALTER TABLE documents ADD COLUMN IF NOT EXISTS chunk_method VARCHAR(50);

-- 更新状态值
UPDATE documents SET parse_status = 'parsed' WHERE parse_status = 'processing';
```

详见：`backend/migrations/add_chunk_method.sql`
